## 一、产品目标

### 一句话描述

将一台闲置的 Android 旧手机改造为无线 CarLife 转接盒，让用户无需额外购买专用硬件即可实现「手机B无线投屏到 WinCE 车机」。

### 可量化成功指标

| 指标 | 目标值 | 优先级 |
|------|--------|--------|
| 视频流端到端延迟 | < 200ms | P0 |
| 全流程连接耗时（从启动到车机出现画面）| < 60s | P0 |
| 连续运行稳定性（无断连） | >= 30 分钟 | P0 |
| 音视频同步误差 | < 100ms | P1 |
| APP 体积 | < 20MB | P2 |
| 支持的旧手机 Android 版本 | Android 6.0+ | P0 |

## 二、目标用户

### 用户画像

**主要用户：「老车机改造者」**

- **年龄段**: 25–45 岁，有一定 Android 使用经验
- **设备现状**:
  - 拥有一台搭载 WinCE 系统的老款车机（CarLife 3.1.0.0）
  - 拥有一台闲置 Android 旧手机（可作为转接盒）
  - 日常主力手机（手机B）安装了 CarWith
- **核心痛点**:
  - 老车机不支持无线 CarLife，必须插 USB 线才能投屏
  - 商业无线盒子（CarlinKit、君用等）需花费 150–500 元
  - 旧手机躺在抽屉里没有二次利用价值
- **技术门槛预期**: 愿意按步骤配置，但不会看代码；期望「傻瓜式」操作

## 三、用户故事

### 核心场景（主流程）

**场景 US-01：首次完整投屏**

> 作为一名拥有 WinCE 车机和闲置旧手机的车主，
> 我希望通过一个 APP 将旧手机变成无线 CarLife 转接盒，
> 以便在不购买额外硬件、不插线的情况下，在车机上看到手机B的导航/音乐界面。

**验收场景**：
1. 旧手机安装并启动 CarLifeWirelessBox APP
2. 手机B已开启热点，旧手机连接热点后 APP 自动发现并识别手机B的 CarWith 服务
3. 旧手机通过 USB 数据线连接车机，APP 引导用户开启 USB 网络共享
4. 车机 CarLife 自动发现并连接转接盒，车机屏幕出现手机B的投屏画面
5. 用户可在车机屏幕上触摸操作手机B的界面，手势正确传回

---

**场景 US-02：断线自动重连**

> 作为一名日常通勤用户，
> 我希望转接盒能在连接中断后（如 Wi-Fi 抖动、USB 插拔）自动恢复连接，
> 以便不必每次都手动重新操作。

---

**场景 US-03：查看连接状态**

> 作为一名普通用户，
> 我希望随时能看到 Wi-Fi 端（连接手机B）和 USB 端（连接车机）的连接状态，
> 以便在出现问题时快速判断是哪一侧断线。

## 四、需求池

### P0 — 必须实现（MVP 不可缺失）

| 编号 | 需求 | 验收标准 |
|------|------|---------|
| P0-01 | **Wi-Fi 客户端连接（HU 角色）** | APP 连接手机B热点后，能以 HU（车机）身份向手机B的 CarWith 发起 6 条 TCP 连接（端口 7240/8240/9240/9241/9242/9340），握手成功 |
| P0-02 | **USB 网络共享引导** | APP 检测到 USB 已连接但未开启网络共享时，弹出提示并跳转到系统设置页；车机通过 USB 获得 IP 地址（192.168.42.x 网段） |
| P0-03 | **TCP 服务端监听（MD 角色）** | APP 在本地 6 个端口（7200/8200/9200/9201/9202/9300）监听车机连接，车机 CarLife 3.1.0.0 能成功连接 |
| P0-04 | **CarLife 协议握手（双角色）** | 完整实现 HU 侧握手（发送协议版本 1.0、HU_INFO、VIDEO_ENCODER_INIT、VIDEO_ENCODER_START）和 MD 侧握手（响应车机的握手消息）；握手完成率 100% |
| P0-05 | **视频流双向透传** | H.264 视频帧从手机B经旧手机透传至车机，车机屏幕正常显示画面，端到端延迟 < 200ms，无持续黑屏 |
| P0-06 | **协议版本强制锁定为 1.0** | APP 发出的所有 HU_PROTOCOL_VERSION 消息中 majorVersion=1, minorVersion=0；不允许用户修改版本号 |
| P0-07 | **认证绕过** | 收到 HU_AUTHEN_REQUEST 时直接返回 authenResult=true，不执行加密计算 |
| P0-08 | **视频分辨率强制 800x480 / 30fps** | VIDEO_ENCODER_INIT 消息中 width=800, height=480, frameRate=30；适配 WinCE 车机解码能力 |

---

### P1 — 重要功能（阶段3完成）

| 编号 | 需求 | 验收标准 |
|------|------|---------|
| P1-01 | **音频流透传（MEDIA + TTS 通道）** | PCM 48kHz/16bit/立体声音频从手机B透传至车机，车机喇叭可正常播放导航语音和媒体音乐，音视频同步误差 < 100ms |
| P1-02 | **触摸事件回传（CTRL 通道）** | 车机屏幕触摸事件（DOWN/UP/MOVE/单击/双击/长按/硬按键）经旧手机转发至手机B，手机B界面响应正确 |
| P1-03 | **断线自动重连** | Wi-Fi 连接中断后，30 秒内自动重新连接手机B；USB 端车机断开后，10 秒内重新建立监听并等待车机重连 |
| P1-04 | **主界面状态实时展示** | 界面显示：Wi-Fi 端状态（未连接/连接中/已连接/错误）、USB 端状态（未连接/连接中/已连接/错误）、已运行时长 |
| P1-05 | **特性配置响应** | 收到车机 FEATURE_CONFIG_REQUEST 时，回复 key="FOCUS_UI", value=1 |
| P1-06 | **前台 Service 保活** | APP 以前台 Service 运行，显示持久通知（连接状态摘要），防止系统杀进程 |

---

### P2 — 优化项（阶段4及后续）

| 编号 | 需求 | 验收标准 |
|------|------|---------|
| P2-01 | **延迟优化** | 通过调整 Socket 缓冲区、零拷贝转发，视频流延迟稳定在 < 150ms |
| P2-02 | **动态比特率调节** | 检测 Wi-Fi 信号强度，弱信号时自动降低视频比特率（从 3Mbps 降至 1.5Mbps），避免卡顿 |
| P2-03 | **VR 通道透传（语音识别）** | 车机麦克风数据（MSG_VR_MIC_DATA）转发至手机B，支持语音识别功能 |
| P2-04 | **崩溃/错误上报** | 握手失败、连接超时等关键错误写入本地日志文件（最多保留 7 天），便于调试 |
| P2-05 | **开机自启动** | 检测到设备充电且 USB 连接时，自动启动 Service 并尝试连接 |
| P2-06 | **多车机兼容性适配** | 提供「兼容模式」选项，支持修改分辨率（800x480 / 1024x600 / 自定义）和帧率（25/30fps） |

## 五、UI 设计稿：主界面状态机

> 说明：这是纯技术工具 APP，界面极简，仅一个主 Activity + 必要的提示对话框。

### 主界面状态机（Mermaid）

```mermaid
stateDiagram-v2
    [*] --> 初始化 : APP 启动

    初始化 --> Wi-Fi检测中 : 检查热点/网络连接

    Wi-Fi检测中 --> 等待Wi-Fi : 未连接到手机B热点
    Wi-Fi检测中 --> 连接手机B中 : 已连接到正确热点 (手机B IP已获取)

    等待Wi-Fi --> 连接手机B中 : 用户手动连接热点后 APP 检测到热点

    连接手机B中 --> Wi-Fi已连接 : 6条TCP连接建立 + CarLife握手成功
    连接手机B中 --> 连接失败_Wi-Fi : 连接超时 / 握手失败

    连接失败_Wi-Fi --> 连接手机B中 : 自动重试 (间隔5s, 最多6次)
    连接失败_Wi-Fi --> 等待Wi-Fi : 超过重试上限，提示用户检查热点

    Wi-Fi已连接 --> USB检测中 : 同步检测USB网络共享状态

    USB检测中 --> 等待USB共享 : USB已连接但网络共享未开启
    USB检测中 --> 监听车机中 : USB网络共享已开启，开始监听6个端口

    等待USB共享 --> 监听车机中 : 用户开启USB网络共享后检测通过

    监听车机中 --> 车机已连接 : 车机CarLife发起连接 + 握手成功
    监听车机中 --> 监听超时 : 60s内无车机连接

    监听超时 --> 监听车机中 : 自动重新监听

    车机已连接 --> 运行中 : Wi-Fi端 + USB端均已连接，开始双向透传

    运行中 --> Wi-Fi断线恢复中 : Wi-Fi连接中断
    运行中 --> USB断线恢复中 : 车机USB断开

    Wi-Fi断线恢复中 --> 运行中 : 重连手机B成功
    Wi-Fi断线恢复中 --> 等待Wi-Fi : 重连失败超上限

    USB断线恢复中 --> 运行中 : 车机重新连接成功
    USB断线恢复中 --> 监听车机中 : 等待车机重新接入

    运行中 --> 已停止 : 用户点击「停止」按钮
    已停止 --> [*] : APP 退出
```

### 主界面元素说明

| 元素 | 说明 |
|------|------|
| Wi-Fi 状态指示灯 | 红/黄/绿三色，对应未连接/连接中/已连接 |
| USB 状态指示灯 | 红/黄/绿三色，对应未连接/连接中/已连接 |
| 连接详情文本 | 显示手机B的 IP 地址、车机 IP 地址、已运行时长 |
| 「开始/停止」按钮 | 单一主操作按钮，切换 Service 运行状态 |
| 错误提示区域 | 在状态机进入失败态时显示简短错误说明和建议操作 |

## 六、待确认问题

以下问题需要在阶段1技术验证中明确，**答案将直接影响核心架构实现**：

| 编号 | 问题 | 影响范围 | 验证方法 |
|------|------|---------|---------|
| Q1 | WinCE 车机是否能识别旧手机的 USB 网卡（RNDIS）？如果不能，USB 通信方式需切换为 AOA 方案 | P0-02、P0-03，整体架构 | 将旧手机开启 USB 网络共享后连接车机 USB，查看车机设备管理器或 CarLife 是否能获得 IP 地址 |
| Q2 | WinCE CarLife 3.1.0.0 通过 TCP 连接旧手机时，目标端口是否为标准 HU 端口（7200/8200/…）？ | P0-03，端口映射是否需要 ADB Forward | 用 Wireshark/tcpdump 抓包，分析车机发起的 TCP 目标端口 |
| Q3 | 手机B（CarWith）以 Wi-Fi 模式发布服务时，具体监听端口是否与 ADB Forward 模式相同（7240/8240/…）？ | P0-01，Wi-Fi TCP 端口号 | 在手机B开启 CarWith 热点后，抓包分析监听端口 |
| Q4 | 认证流程在 6.7.2 版本中是否可完全绕过（直接返回 true），还是存在超时断连风险？ | P0-07 | 参考 CarProjection 项目实测，观察跳过认证后连接是否稳定 |
| Q5 | 旧手机作为 USB 网络共享的「服务端」时，其自身的 IP 地址是否固定为 192.168.42.129（Android RNDIS 默认值）？车机是否能正确 DHCP 获取地址？ | P0-02，车机连接旧手机的目标 IP 发现机制 | 开启 USB 网络共享后，查看旧手机 usb0 网口 IP 及车机分配到的 IP |
| Q6 | WinCE 车机屏幕分辨率是否确为 800x480？如不是，VIDEO_ENCODER_INIT 参数是否需要动态协商？ | P0-08 | 查看车机规格手册，或通过抓包分析车机发出的 VIDEO_ENCODER_INIT 请求参数 |
| Q7 | 旧手机同时持有 Wi-Fi 连接（连手机B热点）和 USB 网络共享（连车机）时，系统路由表是否会冲突？TCP 数据是否能正确分流？ | P0-01、P0-03，网络路由稳定性 | 建立两侧连接后，分别 ping 手机B IP 和车机 IP，确认双向可达 |
