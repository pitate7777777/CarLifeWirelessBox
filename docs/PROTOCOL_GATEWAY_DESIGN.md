# CarLife 协议网关设计方案

> **状态**: 备用计划（长远方案）
> **创建日期**: 2026-05-13
> **优先级**: P2（当前主方案为 CarWith 旧版直连）
> **触发条件**: CarWith 停止维护、新版 CarLife 成为主流、或用户无法获取 CarWith 时启用

---

## 1. 问题背景

### 1.1 当前方案的限制

当前 CarLifeWirelessBox 要求手机B使用 **CarWith 或 CarLife 6.7.2 以下版本**，因为两端都锁定协议版本 1.0。

```
[手机B - CarWith]  ←协议1.0→  [旧手机APP]  ←协议1.0→  [WinCE车机]
     旧版协议                      双角色桥接                  CarLife 3.1.0.0
```

**限制**：
- 用户必须在手机B上安装特定版本的 CarWith
- CarWith 可能停止维护或从应用商店下架
- 新手机出厂预装的可能是新版 CarLife（7.2+），无法降级
- 部分手机品牌（如小米）的 CarWith 可能有定制差异

### 1.2 目标

解除版本限制，让手机B可以使用**任意版本**的 CarLife / CarWith：

```
[手机B - 新版CarLife]  ←新版协议→  [旧手机APP - 协议网关]  ←旧版协议1.0→  [WinCE车机]
     7.2+                                   翻译层                              3.1.0.0
```

## 2. 协议差异分析（待逆向确认）

### 2.1 已知信息

| 维度 | 旧版 (≤6.7.2) | 新版 (≥7.2) | 来源 |
|------|---------------|-------------|------|
| 协议版本号 | 1.0 | **未知** | 实测可连 WinCE |
| 认证 | 可绕过 | **可能强制** | CarProjection 代码 |
| 视频编码 | H.264, 800×480 | **可能 H.265** | 用户反馈黑屏 |
| 音频编码 | PCM 48kHz/16bit | **可能 Opus** | 推测 |
| 包头格式 | CMD=8B, Media=12B | **可能变化** | CarLifeVehicleLib |

### 2.2 黑屏原因推测

CarLife 7.2+ 连接 WinCE 车机黑屏，可能原因（按可能性排序）：

1. **协议版本号不匹配**（高概率）
   - 新版发送 `majorVersion=2`，WinCE 车机（3.1.0.0）不识别，断开连接
   - 表现：连接在版本协商阶段断开

2. **认证机制加强**（中概率）
   - 新版要求真正的加密认证，绕过认证会导致后续数据被拒绝
   - 表现：连接在认证阶段断开或黑屏

3. **视频编码格式不兼容**（中概率）
   - 新版默认输出 H.265，WinCE 车机只能解码 H.264
   - 表现：连接成功但画面黑屏

4. **特性配置新增必选项**（低概率）
   - 新版 `FEATURE_CONFIG` 新增了 WinCE 车机不支持的必选特性
   - 表现：连接在特性协商阶段断开

## 3. 网关架构设计

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                    CarLifeWirelessBox                        │
│                    协议网关层 (Protocol Gateway)              │
│                                                             │
│  ┌───────────────────┐           ┌───────────────────┐     │
│  │ HuRole (新版协议)  │           │ MdRole (旧版1.0)  │     │
│  │                   │           │                   │     │
│  │ CMD  ←──翻译──→   │  Stream   │ CMD  (锁死1.0)    │     │
│  │ VIDEO ←──转码──→  │  Bridge   │ VIDEO (H.264)     │     │
│  │ MEDIA ←──转码──→  │  ←数据→   │ MEDIA (PCM)       │     │
│  │ TTS   ←──直通──→  │           │ TTS               │     │
│  │ VR    ←──直通──→  │           │ VR                │     │
│  │ CTRL  ←──直通──→  │           │ CTRL              │     │
│  └───────────────────┘           └───────────────────┘     │
│           │                              │                  │
│     连接手机B CarWith              等待WinCE车机连接          │
│     (新版协议握手)                 (旧版协议握手)             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 翻译层分类

根据通道特性，翻译策略分为三类：

| 翻译类型 | 通道 | 策略 | 说明 |
|---------|------|------|------|
| **协议翻译** | CMD | 拦截+重写 | 版本号、设备信息、认证、特性配置 |
| **格式转码** | VIDEO, MEDIA | 解码+重编码 | H.265→H.264, Opus→PCM |
| **直通透传** | TTS, VR, CTRL | 直接转发 | 数据格式在版本间不变 |

### 3.3 CMD 通道翻译流程

```
手机B → [新版CMD消息] → 解析 → 翻译规则 → 构造旧版CMD消息 → 车机
                                                              |
车机 → [旧版CMD消息] → 解析 → 翻译规则 → 构造新版CMD消息 → 手机B
```

#### 翻译规则表

| 消息ID | 方向 | 翻译规则 |
|--------|------|---------|
| `HU_PROTOCOL_VERSION` | B→车 | **重写版本号**：major=2→1, minor=0→0 |
| `VERSION_MATCH_STATUS` | 车→B | 直通（匹配状态不变） |
| `HU_INFO` | B→车 | **过滤字段**：只保留旧版 23 个字段 |
| `MD_INFO` | 车→B | **补充字段**：按新版格式填充新增字段 |
| `HU_AUTHEN_REQUEST` | B→车 | **可能需要**：转发认证数据或重新构造 |
| `MD_AUTHEN_RESPONSE` | 车→B | **可能需要**：将旧版响应包装为新版格式 |
| `HU_AUTHEN_RESULT` | B→车 | 直通 |
| `MD_AUTHEN_RESULT` | 车→B | 直通 |
| `FEATURE_CONFIG_REQUEST` | 车→B | **补充字段**：按新版格式填充 |
| `FEATURE_CONFIG_RESPONSE` | B→车 | **过滤字段**：只保留旧版支持的特性 |
| `VIDEO_ENCODER_INIT` | B→车 | **降级参数**：H.265→H.264, 720P→480P |
| `VIDEO_ENCODER_INIT_DONE` | 车→B | **升级参数**：H.264→目标格式 |
| `VIDEO_ENCODER_START` | B→车 | 直通 |

### 3.4 视频转码管线

```
                    ┌─────────────────────────────────┐
                    │        视频转码管线               │
                    │                                 │
手机B ──H.265──→ │ MediaCodec 解码器 (H.265)       │
                    │         ↓ Surface               │
                    │ MediaCodec 编码器 (H.264)       │
                    │         ↓                       │
                    │ H.264 NAL Units                 │
WinCE车机 ←H.264── │         ↓                       │
                    │ SPS/PPS 拼接 + 帧封装           │
                    └─────────────────────────────────┘
```

**实现要点**：
- 使用 `MediaCodec` 异步模式（`AsyncCallback`）
- 解码器输出 Surface 直连编码器输入 Surface（零拷贝）
- 关键帧前拼接缓存的 SPS/PPS
- 如果手机B也输出 H.264，跳过转码，直通即可

**性能要求**：
- 旧手机需支持 H.265 硬解码（Android 5.0+ 大部分支持）
- 旧手机需支持 H.264 硬编码（所有 Android 设备支持）
- 延迟预算：解码 ~8ms + 编码 ~8ms = ~16ms（可接受）

### 3.5 音频转码管线

```
手机B ──Opus/AAC──→ MediaCodec 解码器 → PCM 48kHz/16bit/立体声 → WinCE车机
```

- 如果手机B输出 PCM，直通即可
- Opus→PCM 转码开销很小（CPU 可胜任）

### 3.6 动态版本检测

网关需要自动检测手机B的协议版本，选择翻译或直通模式：

```kotlin
enum class GatewayMode {
    PASSTHROUGH,    // 直通模式（手机B也是旧版协议，无需翻译）
    TRANSLATION,    // 翻译模式（手机B是新版协议，需要翻译）
    AUTO            // 自动检测
}

fun detectPhoneBVersion(huProtocolVersion: CarlifeProtocolVersion): GatewayMode {
    return if (huProtocolVersion.majorVersion <= 1) {
        GatewayMode.PASSTHROUGH  // 旧版协议，直通
    } else {
        GatewayMode.TRANSLATION  // 新版协议，需要翻译
    }
}
```

## 4. 实现计划

### 4.1 前置研究任务（Phase 0）

**必须在实现前完成**，否则无法确定翻译规则。

| 研究任务 | 方法 | 产出 | 预计耗时 |
|---------|------|------|---------|
| 逆向新版 CarLife APK (7.2+) | jadx 反编译 | 新版 proto 定义、握手流程 | 2-3天 |
| 抓包分析新版协议 | Wireshark + 手机B热点 | 消息ID、版本号、包头格式 | 1-2天 |
| 确认认证算法 | 分析 `CarLifeSenderService` | 是否可绕过、加密算法 | 1-2天 |
| 确认视频编码默认格式 | 抓包分析 `VIDEO_ENCODER_INIT` | H.264 vs H.265 | 0.5天 |
| 测试新版直连WinCE | 实机测试 | 确认黑屏的具体阶段 | 0.5天 |

**预计总耗时**：1-2周

### 4.2 实现阶段

| 阶段 | 任务 | 预计耗时 | 依赖 |
|------|------|---------|------|
| **Phase 0** | 逆向分析 + 协议差异确认 | 1-2周 | 无 |
| **Phase 1** | CMD 通道协议翻译 | 1-2周 | Phase 0 |
| **Phase 2** | 视频格式检测 + 按需转码 | 2-3周 | Phase 0 |
| **Phase 3** | 音频格式检测 + 按需转码 | 1周 | Phase 0 |
| **Phase 4** | 动态版本检测 + 自动模式切换 | 1周 | Phase 1-3 |
| **Phase 5** | 兼容性测试 + 优化 | 持续 | Phase 4 |

**预计总开发时间**：6-10周（不含 Phase 0 研究）

### 4.3 涉及的代码改动

| 模块 | 改动 | 影响范围 |
|------|------|---------|
| `HuRole.kt` | 支持新版协议版本号、新认证流程 | `connect()`, `sendProtocolVersion()`, `sendAuthenRequest()` |
| `ProtocolTranslator.kt` | 从空壳变为真正的翻译器 | 新增 `translateCmd()`, `translateVideo()`, `translateAudio()` |
| `VersionDetector.kt` | 动态版本检测 | 新增 `detectFromHandshake()` |
| `ConnectionService.kt` | 网关模式管理 | 新增 `GatewayMode` 配置 |
| `MdRole.kt` | 无需改动（锁死1.0） | — |
| `Channel.kt` | 可能需要支持新包头格式 | `readCarLifeMsg()`, `sendCarLifeMsg()` |
| `SettingsActivity.kt` | 新增网关模式开关 | UI |

## 5. 风险与降级策略

### 5.1 风险评估

| 风险 | 影响 | 概率 | 应对 |
|------|------|------|------|
| 新版协议结构性变更（包头格式完全不同） | 🔴 致命 | 低 | 需要完全重写 Channel 层 |
| 认证算法无法逆向或无法绕过 | 🔴 致命 | 中 | 尝试中间人代理方式 |
| H.265→H.264 转码旧手机性能不足 | 🟡 高 | 中 | 降级到只支持 H.264 手机B |
| 新版 CarLife 使用 TLS/加密通道 | 🔴 致命 | 低 | 需要 hook SSL 层 |
| 新版 CarLife 使用 gRPC 替代自定义协议 | 🔴 致命 | 极低 | 完全重写，工作量极大 |

### 5.2 降级策略

**策略A：混合模式**
- CMD 通道做协议翻译（版本号、认证）
- 媒体通道做格式转码（H.265→H.264）
- 其他通道直通
- 适用：协议差异可控

**策略B：中间人代理**
- 旧手机同时模拟新版 CarLife 客户端（对手机B）和旧版 CarLife 服务端（对车机）
- 两端独立完成握手，中间做数据转发
- 适用：认证无法绕过

**策略C：SDK Hook**
- 如果旧手机有 Root 权限，通过 Xposed/LSPosed hook CarLife SDK 的协议层
- 直接修改 SDK 内部的版本号和认证逻辑
- 适用：极端情况，作为最后手段

## 6. 与当前方案的关系

```
                     ┌──────────────────────┐
                     │    选择决策树          │
                     └──────────┬───────────┘
                                │
                    ┌───────────▼───────────┐
                    │ CarWith 可用且稳定？    │
                    └───────────┬───────────┘
                          │           │
                         YES          NO
                          │           │
                          ▼           ▼
                  ┌──────────┐  ┌──────────────┐
                  │ 当前方案  │  │ 协议网关方案  │
                  │ (主方案)  │  │ (备用方案)    │
                  │ 稳定可靠  │  │ 需要逆向研究  │
                  └──────────┘  └──────────────┘
```

| 维度 | 当前方案 (CarWith旧版) | 协议网关方案 |
|------|----------------------|-------------|
| 手机B要求 | 必须安装 CarWith | 任意版本 CarLife |
| 稳定性 | ✅ 高（协议完全匹配） | ⚠️ 待验证 |
| 开发复杂度 | 🟢 低 | 🔴 高 |
| 维护成本 | 🟢 低 | 🟡 中（需跟踪新版变化） |
| 用户体验 | ⚠️ 需安装特定APP | ✅ 开箱即用 |
| 适用场景 | CarWith 可用时 | CarWith 不可用时 |

**结论**：协议网关作为长远备用计划储备。当 CarWith 不可用、停止维护、或新版 CarLife 成为唯一选择时，启动 Phase 0 逆向研究，评估可行性后决定是否实现。
