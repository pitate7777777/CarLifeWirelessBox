## 一、项目概述

### 1.1 项目目标

将旧Android手机改造为**无线CarLife转接盒**，实现：

```
[手机B - CarWith]     ← 开启热点 + 运行CarWith
       ↓ (Wi-Fi)
[旧手机 - CarLife转接盒]    ← 你的APP (桥接中转)
       ↓ (USB数据线)
[WinCE车机 - CarLife 3.1.0.0] ← 最后一版支持WinCE的CarLife
```

### 1.2 核心功能

| 功能 | 说明 |
|------|------|
| 无线接收 | 旧手机连接手机B的热点，接收CarLife数据流 |
| 有线转发 | 旧手机通过USB连接WinCE车机，转发投屏内容 |
| 视频流 | H.264编码，30fps，延迟 <200ms |
| 音频流 | PCM 48kHz/16bit/立体声 |
| 反向触控 | 车机触摸事件回传到手机B |

### 1.3 关键约束

| 约束 | 说明 |
|------|------|
| 车机系统 | WinCE (非Android) |
| 车机CarLife版本 | 3.1.0.0 (最后WinCE版本) |
| 旧手机 | Android 6.0+，推荐已Root |
| 连接方式 | 手机B开热点，旧手机连接 |

## 三、CarLife协议深度解析

### 3.0 版本兼容性关键发现 ⚠️

#### 版本分界点

| CarLife版本 | WinCE车机兼容性 | 说明 |
|-------------|-------------------|------|
| **6.7.2及之前** | ✅ **完全兼容** | 可成功有线连接WinCE车机 |
| **7.2及之后** | ❌ **黑屏不兼容** | 连接在90%处断开或黑屏 |

#### 影响分析

```
原本架构（有问题）:
[手机B - CarWith (旧版)] → WiFi → [旧手机] → USB → [WinCE车机]
  新协议(8.x)                             旧协议(3.1.0.0)
  ❌ 协议不匹配！

修正架构（推荐）:
[手机B - CarWith] → WiFi → [旧手机] → USB → [WinCE车机]
  旧协议(6.7.2)                              旧协议(3.1.0.0)
  ✅ 协议匹配！
```

#### 7.2版本可能的变化（推测）

基于用户反馈和黑屏现象，7.2版本可能包含以下变化：

| 可能的变化 | 证据 | 影响 |
|-------------|------|------|
| **协议版本号升级** | CarLifeProtocolVersion可能从1.0变为2.0 | WinCE车机(3.1.0.0)无法识别 |
| **认证机制加强** | AuthenRequest/Response可能强制校验 | 绕过认证可能导致黑屏 |
| **视频编码参数变更** | 默认分辨率/帧率可能提高 | WinCE车机无法解码高分辨率 |
| **特性配置强制检查** | FEATURE_CONFIG可能新增必选项 | 缺少某些特性导致连接中断 |
| **USB通信方式变更** | 可能从AOA改为其他模式 | WinCE车机无法识别新的USB模式 |

#### 应对方案

**方案A: 使用6.7.2版本（推荐✅）**
- 手机B安装CarWith
- 旧手机转接盒实现6.7.2协议
- 完全兼容WinCE车机

**方案B: 协议转换（复杂❌）**
- 旧手机同时理解6.7.2和8.x协议
- 在中间做协议转换
- 开发量大，维护困难

**结论**: **必须使用CarWith版本！**

### 3.1 通信架构

CarLife协议采用**6个独立TCP通道**设计：

```
                   ┌── CMD通道   (端口7240) ── 控制信令
                   ├── VIDEO通道 (端口8240) ── 视频流
                   ├── MEDIA通道 (端口9240) ── 媒体音频
手机B(MD) ────────┼── TTS通道   (端口9241) ── 导航语音
                   ├── VR通道    (端口9242) ── 语音识别
                   └── CTRL通道  (端口9340) ── 触摸控制
```

### 3.2 端口映射

**ADB Forward模式** (USB有线连接时，车机端执行):

| 通道 | 手机端(MD)端口 | 车机端(HU)本地端口 | 方向 |
|------|---------------|-------------------|------|
| CMD  | 7240 | 7200 | 双向 |
| VIDEO | 8240 | 8200 | MD→HU |
| MEDIA | 9240 | 9200 | MD→HU |
| TTS  | 9241 | 9201 | MD→HU |
| VR   | 9242 | 9202 | HU→MD |
| CTRL | 9340 | 9300 | HU→MD |

```bash
# 车机端执行的ADB Forward命令
adb forward tcp:7200 tcp:7240   # CMD
adb forward tcp:8200 tcp:8240   # VIDEO
adb forward tcp:9200 tcp:9240   # MEDIA
adb forward tcp:9201 tcp:9241   # TTS
adb forward tcp:9202 tcp:9242   # VR
adb forward tcp:9300 tcp:9340   # CTRL
```

**Wi-Fi直连模式**:
- 直连手机IP的6个端口 (7240/8240/9240/9241/9242/9340)
- 无需ADB Forward

### 3.3 协议包头格式

#### 3.3.1 USB传输层包头 (8字节)

CarProjection项目使用的**单流复用**格式:

```
偏移  字段             类型    说明
─────────────────────────────────────────
0-2   (保留)           -      未使用，值为0
3     msg_type         byte   通道类型标识 (1=CMD,2=VIDEO,3=MEDIA,6=TOUCH)
4-7   payload_length   int32  CarLife消息体长度 (Big-Endian)
```

#### 3.3.2 CMD/CTRL通道消息体 (8字节头 + Protobuf)

```
偏移   字段           类型      说明
──────────────────────────────────────
0-1    data_len      short     Protobuf数据长度 (Big-Endian)
2-3    (保留)         -        未使用
4-7    service_type  int32     服务类型/消息ID (Big-Endian)
8+     protobuf_data  bytes    Protobuf序列化数据
```

#### 3.3.3 VIDEO/MEDIA/TTS/VR通道消息体 (12字节头 + 数据)

```
偏移    字段           类型     说明
──────────────────────────────────────
0-3     data_len      int32    原始数据长度 (Big-Endian)
4-7     timestamp     int32    时间戳 (毫秒)
8-11    service_type  int32    服务类型/消息ID (Big-Endian)
12+     raw_data      bytes    H.264帧/PCM采样
```

#### 3.3.4 各通道缓冲区大小

| 通道 | 包头长度 | 数据缓冲区大小 |
|------|----------|---------------|
| CMD  | 8字节 | 40KB |
| VIDEO | 12字节 | 600KB |
| MEDIA | 12字节 | 100KB |
| TTS  | 12字节 | 50KB |
| VR   | 12字节 | 50KB |
| CTRL | 8字节 | 10KB |

### 3.4 通道类型与消息ID

#### 通道类型枚举

```cpp
CMD_CHANNEL   = 1   // 命令通道
VIDEO_CHANNEL = 2   // 视频通道
MEDIA_CHANNEL = 3   // 媒体音频通道
TTS_CHANNEL   = 4   // 导航语音通道
VR_CHANNEL    = 5   // 语音识别通道
CTRL_CHANNEL  = 6   // 触摸控制通道
```

#### 消息ID编码规则

```
0x0001xxxx = CMD通道消息
  0x000180xx = HU(车机)→MD(手机)
  0x000100xx = MD(手机)→HU(车机)
0x0002xxxx = VIDEO通道消息
0x0003xxxx = MEDIA通道消息
0x0004xxxx = TTS通道消息
0x0005xxxx = VR通道消息
  0x000580xx = HU→MD
0x0006xxxx = CTRL通道消息
  0x000680xx = HU→MD
```

### 3.6 认证机制

**认证是可选的**，可跳过。CarProjection项目中直接返回 `result=true` 绕过认证。

如果需要实现认证:
1. HU发送 `randomValue` (如 `"CarLife_1.0.0"`)
2. MD用内置密钥加密后返回 `encryptValue`
3. HU验证加密结果
4. **加密算法未公开**，需要逆向分析百度CarLife APK

#### 4.1.2 USB通信实现

### 4.2 CarLifeVehicleLib 源码分析 (协议规范)

#### 4.2.1 核心发现

1. **USB通信依赖ADB**: 车机端执行 `adb forward` 建立端口转发，然后通过TCP Socket通信

4. **不包含解码器**: 库只负责数据传输，解码由上层实现
5. **Wi-Fi直连支持**: `connectionSetup(string mdIP)` 方法直接连接手机IP

#### 4.2.2 连接模式

```cpp

connectionSetup(mdIP, interfaceName);  // 多网卡场景
```

### 5.3 USB连接方式分析

**关键问题**: 旧手机通过USB连接WinCE车机，使用什么协议？

| 方式 | 原理 | 旧手机角色 | 可行性 |
|------|------|-----------|--------|
| **AOA Accessory** | 手机作为USB Device，车机作为USB Host | Accessory | ⚠️ 需要车机支持AOA |
| **ADB Forward** | 手机开ADB，车机通过ADB端口转发 | ADB Device | ❌ WinCE不支持ADB |
| **USB Gadget** | Root后通过ConfigFS模拟USB设备 | 任意USB设备 | ✅ 需要Root |
| **USB Tethering** | 手机USB共享网络，通过TCP通信 | RNDIS设备 | ✅ 旧手机原生支持 |

**推荐方案**: USB Tethering (USB网络共享)

```
旧手机开启USB网络共享
    ↓
WinCE车机通过USB识别为网卡
    ↓
车机CarLife通过TCP连接旧手机IP
    ↓
旧手机APP作为桥接，转发数据到手机B
```

**优势**:
- 不需要Root
- 利用Android原生USB网络共享功能
- WinCE端CarLife通过TCP连接（CarLife天然支持TCP）
- 开发量最小
- 协议完全兼容（都是6.7.2旧版）

**劣势**:
- 需要验证WinCE车机是否能识别USB网卡
- 延迟可能略高于直连

### 5.5 最终技术路线详细设计 (方案A)

```
                    Wi-Fi (TCP 6端口)                USB Tethering (TCP 6端口)
[手机B] ←────────────→ [旧手机 - 桥接APP] ←────────────→ [WinCE车机]
   CarWith              双角色桥接                   CarLife 3.1.0.0
                                    
   开启热点                    1. 连接手机B热点               USB共享网络
   IP: 192.168.x.x              2. 以HU身份连接手机B            车机获得IP: 192.168.42.x
                             3. 以MD身份监听本地端口        连接旧手机IP
                             4. 双向数据转发
```

**旧手机APP的完整工作流程**:

1. **启动阶段**:
   - 连接手机B的Wi-Fi热点 (192.168.x.x)
   - 记录手机B的IP地址
   - 开启USB网络共享 (设置 → 网络共享 → USB共享)
   - 等待WinCE车机通过USB网络连接

2. **连接手机B (HU角色模拟)**:
   - 建立6个TCP连接到手机B:
     - CMD: 手机B_IP:7240
     - VIDEO: 手机B_IP:8240
     - MEDIA: 手机B_IP:9240
     - TTS: 手机B_IP:9241
     - VR: 手机B_IP:9242
     - CTRL: 手机B_IP:9340
   - 发送握手消息:
     - `0x00018001` HU_PROTOCOL_VERSION (majorVersion=1, minorVersion=0)
     - `0x00018003` HU_INFO (os=android, board=..., ...)
     - `0x00018007` VIDEO_ENCODER_INIT (width=800, height=480, frameRate=30)
     - `0x00018009` VIDEO_ENCODER_START
   - 触发手机B开始投屏

3. **等待车机连接 (MD角色模拟)**:
   - 在6个本地端口监听:
     - CMD: 0.0.0.0:7200
     - VIDEO: 0.0.0.0:8200
     - MEDIA: 0.0.0.0:9200
     - TTS: 0.0.0.0:9201
     - VR: 0.0.0.0:9202
     - CTRL: 0.0.0.0:9300
   - 等待WinCE车机CarLife连接 (车机会主动连接)
   - 响应车机的握手消息:
     - 收到 `0x00018001` → 回复 `0x00010002` (matchStatus=1)
     - 收到 `0x00018003` → 回复 `0x00010004` (发送手机虚拟信息)
     - 收到 `0x00018007` → 回复 `0x00010008` (INIT_DONE)
     - 收到 `0x00018009` → 开始转发视频流

4. **数据桥接 (核心)**:
   - **手机B → 旧手机 → 车机**:
     - 视频流: 从手机B VIDEO通道读取 → 写入车机VIDEO通道
     - 音频流: 从手机B MEDIA/TTS通道读取 → 写入车机MEDIA/TTS通道
   - **车机 → 旧手机 → 手机B**:
     - 触摸事件: 从车机CTRL通道读取 → 写入手机B CTRL通道
     - 语音数据: 从车机VR通道读取 → 写入手机B VR通道

### 6.5 协议包头封装工具

```java
public class CarLifeProtocol {
    // CMD/CTRL通道包头封装
    public static byte[] packCmdMessage(int serviceType, byte[] protobufData) {
        byte[] payload = new byte[8 + (protobufData != null ? protobufData.length : 0)];
        // 字节0-1: 数据长度
        shortToBytes((short)(protobufData != null ? protobufData.length : 0), payload, 0);
        // 字节4-7: 服务类型
        intToBytes(serviceType, payload, 4);
        // 字节8+: Protobuf数据
        if (protobufData != null) {
            System.arraycopy(protobufData, 0, payload, 8, protobufData.length);
        }
        return payload;
    }

    // VIDEO/MEDIA通道包头封装
    public static byte[] packMediaMessage(int serviceType, byte[] rawData) {
        byte[] payload = new byte[12 + rawData.length];
        // 字节0-3: 数据长度
        intToBytes(rawData.length, payload, 0);
        // 字节4-7: 时间戳
        intToBytes((int)System.currentTimeMillis(), payload, 4);
        // 字节8-11: 服务类型
        intToBytes(serviceType, payload, 8);
        // 字节12+: 原始数据
        System.arraycopy(rawData, 0, payload, 12, rawData.length);
        return payload;
    }

    // USB传输层包头封装 (单流复用模式)
    public static byte[] packUsbHeader(byte msgType, int payloadLength) {
        byte[] header = new byte[8];
        header[3] = msgType;           // 通道类型
        intToBytes(payloadLength, header, 4);  // 消息体长度
        return header;
    }
}
```
