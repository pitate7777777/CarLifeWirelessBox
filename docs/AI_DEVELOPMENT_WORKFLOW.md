# AI 辅助开发流程指南

> 以 CarLife Wireless Box 项目为实践总结，适用于 Android/Kotlin 及类似中型项目。

---

## 一、总体流程

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 1. 上下文 │───→│ 2. 生成  │───→│ 3. 审查  │───→│ 4. 验证  │───→│ 5. 提交  │
│   构建   │    │   代码   │    │   修复   │    │   测试   │    │   追溯   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
```

核心原则：**AI 是副驾驶，方向盘在你手里。**

---

## 二、阶段详解

### 阶段 1：上下文构建

**目标**：让 AI 理解你的项目，而不是从零猜测。

#### 1.1 项目级上下文文件

在项目根目录维护以下文件：

| 文件 | 用途 | 更新频率 |
|------|------|----------|
| `AGENTS.md` | AI 专属上下文（架构、规范、检查清单） | 架构变更时 |
| `docs/DEVELOPMENT.md` | 完整开发指南（协议、线程模型、常见问题） | 每个迭代 |
| `.editorconfig` | 代码风格 | 一次性 |

#### 1.2 AGENTS.md 模板

```markdown
# AGENTS.md — AI 开发上下文

## 项目信息
- 语言/框架/版本
- 构建工具
- 核心协议/规范

## AI 开发检查清单
生成代码后，自查以下项目：
- [ ] 平台兼容性检查（API 版本差异）
- [ ] 资源生命周期（socket/流/锁）
- [ ] 编解码器/协议类型一致性
- [ ] 线程安全（锁粒度、竞态条件）

## 关键文件
| 文件 | 用途 |
|------|------|
| src/.../Core.kt | 核心逻辑 |

## 常见陷阱
1. 不要同时连接同一组端口
2. 未转码时保留原始类型
```

#### 1.3 对话级上下文

每次对话开始时，告诉 AI：

```
我在开发 [项目名]，这是一个 [一句话描述]。
请先读 AGENTS.md 和 docs/DEVELOPMENT.md 了解项目。
当前任务：[具体需求]
```

**不要假设 AI 记得上次对话的内容。每次都是新对话。**

### 阶段 2：代码生成

#### 2.1 需求描述规范

| 差 | 好 |
|---|---|
| "帮我写个视频编码器" | "用 MediaCodec H.264 硬编码，输入是 MediaProjection 的 Surface，输出 Annex-B NAL，支持动态码率" |
| "修复这个 bug" | "握手卡在第 4 步，日志显示 MD_AUTHEN_RESPONSE 未收到，可能原因：1) 通道未连接 2) 超时太短" |
| "优化性能" | "getLocalIpAddress() 调用了两次 getNetworkInterfaces()，合并为单次遍历" |

#### 2.2 分模块生成

```
❌ "帮我写完整个项目"
✅ "先定义 Channel 接口 → 再实现 TcpChannel → 再写 ChannelHeader 序列化"
```

**原因**：
- 一次性生成 → 模块间接口不一致，状态管理混乱
- 分步生成 → 每步可验证，接口逐步收敛

#### 2.3 生成后立即自问

AI 输出代码后，检查：

1. **平台兼容性**：这个 API 在 minSdk 上可用吗？
2. **资源泄漏**：Socket/Stream/Cursor 关了吗？
3. **线程安全**：多线程访问的变量用 Atomic 或锁了吗？
4. **错误处理**：异常被捕获并记录了吗？
5. **类型一致性**：返回的类型和实际数据匹配吗？

---

### 阶段 3：审查修复

**这是最关键的阶段。AI 写的代码必须审查。**

#### 3.1 审查清单

```bash
# 运行自动检查
bash scripts/check.sh
```

手动审查重点：

| 检查项 | 方法 | 典型问题 |
|--------|------|----------|
| 废弃 API | grep "Build.SERIAL\|@Suppress" | API 29+ 返回 null |
| 资源泄漏 | grep "Socket()\|DatagramSocket()\|FileWriter" | 未用 use{} 包裹 |
| 类型错误 | 检查函数返回类型与实际数据 | 编解码器类型标注错误 |
| 重复代码 | 检查相似函数 | getNetworkInterfaces() 调两次 |
| 线程安全 | 检查 @Volatile / synchronized | 读写锁不匹配 |
| 魔数 | grep "0x\|[0-9][0-9][0-9][0-9]" | 未命名的常量 |

#### 3.2 常见 AI 生成错误模式

| 模式 | 示例 | 修复 |
|------|------|------|
| **类型欺骗** | 未转码却返回目标 codec 类型 | 返回原始类型 |
| **平台盲区** | 使用高版本 API 无降级 | 加版本检查 + 兜底 |
| **资源泄漏** | new Socket() 后异常未关闭 | 用 use{} 或 try-finally |
| **重复逻辑** | 同一个网络查询写两遍 | 合并为单次调用 |
| **锁竞争** | synchronized(this) 覆盖读写 | 用独立锁对象 |
| **静默失败** | catch 块为空或只 printStackTrace | 用 LogUtils 记录上下文 |

#### 3.3 AI 辅助审查

让 AI 审查自己的代码：

```
请审查以上代码，重点关注：
1. 平台兼容性（minSdk 26）
2. 资源泄漏
3. 线程安全
4. 与现有代码风格的一致性
列出所有问题，按严重程度排序。
```

### 阶段 4：验证

#### 4.1 单元测试

```bash
./gradlew testDebugUnitTest
```

**原则**：
- 每个工具类必须有测试
- 协议序列化/反序列化必须有测试
- 边界条件（空值、负数、超大值）必须覆盖

**AI 帮写测试的优势**：AI 会从"这个函数应该怎么用"的角度思考，帮你发现接口设计问题。

#### 4.2 代码检查

```bash
bash scripts/check.sh
```

检查项：
- Build.SERIAL 使用
- println / printStackTrace
- 测试覆盖率
- wildcard import

#### 4.3 集成验证

**AI 无法替你做的事**：
- 真机运行
- 网络环境测试
- 性能 profiling
- 安全审计

这些必须人工完成。

### 阶段 5：提交追溯

#### 5.1 提交规范

```
<type>: <简短描述>

<详细说明（可选）>

变更文件列表
```

type 类型：
- `fix`: 修复 bug
- `feat`: 新功能
- `refactor`: 重构
- `chore`: 工具/文档/配置
- `test`: 测试
- `docs`: 文档

#### 5.2 AI 改动单独提交

每次 AI 改完代码，单独提交，写清楚改了什么。这样出问题可以快速定位是哪次 AI 改动引入的。

```bash
# 好：分两次提交
git commit -m "fix: Build.SERIAL 废弃修复"
git commit -m "test: 补充 LogFileManager 单元测试"

# 差：混在一起
git commit -m "一堆修改"
```

#### 5.3 变更追溯

如果发现问题，用 `git log` 和 `git bisect` 定位：

```bash
# 查看 AI 相关的提交
git log --oneline --grep="fix:\|refactor:\|chore:"

# 二分查找引入 bug 的提交
git bisect start
git bisect bad HEAD
git bisect good <已知好的提交>
```

## 三、工具链

### 项目级工具

| 工具 | 用途 | 命令 |
|------|------|------|
| `scripts/check.sh` | 代码质量自动检查 | `bash scripts/check.sh` |
| `detekt` | Kotlin 静态分析 | `./gradlew detekt` |
| `.editorconfig` | 代码风格统一 | 编辑器自动应用 |
| JUnit + Robolectric | 单元测试 | `./gradlew testDebugUnitTest` |

### AI 使用技巧

| 场景 | 做法 |
|------|------|
| 写新模块 | 先给 AI 看接口定义和相邻模块，再让它写 |
| 修 bug | 给 AI 错误日志 + 相关代码 + 预期行为 |
| 重构 | 让 AI 列出改动计划，确认后再执行 |
| 代码审查 | 让 AI 按清单逐项检查 |
| 写测试 | 给 AI 函数签名 + 边界条件清单 |

## 四、反模式（避免）

| 反模式 | 为什么不行 | 正确做法 |
|--------|-----------|----------|
| 盲信 AI 输出 | 没有运行环境，无法验证 | 必须审查 + 测试 |
| 一次生成整个项目 | 接口不一致，状态混乱 | 分模块逐步生成 |
| 不写上下文文件 | 每次对话重新解释 | 维护 AGENTS.md |
| AI 改完直接提交 | 出问题无法定位 | 单独提交，写清描述 |
| 只让 AI 写代码 | 测试和文档同样重要 | 让 AI 写测试 + 文档 |
| 忽略检查脚本警告 | 小问题积累成大问题 | 零容忍，立即修 |

## 五、实战案例

### 案例：CarLife 项目代码审查

**输入**：一个 40 个源文件的 Android 项目

**AI 做了什么**：
1. 读完全部源码（~5000 行）
2. 发现 7 个 bug：
   - Build.SERIAL 废弃（API 29+ 返回 null）
   - 编解码器类型标注错误（H265→H264, Opus→AAC）
   - 采样率常量不一致（48000 vs 44100）
   - 网络接口双重遍历
   - 整数除法精度丢失
   - DatagramSocket 资源泄漏
   - 测试断言与实现不匹配
3. 修复全部问题
4. 补充 4 个测试文件
5. 编写开发文档 + 质量检查脚本

**人做了什么**：
- 提供项目仓库
- 审查 AI 的修复方案
- 确认提交

**教训**：
- AI 找到的 bug 中，编解码器类型错误是最隐蔽的——代码逻辑完全正确，只是返回值语义错误
- 测试和实现不匹配说明之前的 AI 生成代码时没有保持一致性
- 质量检查脚本能防止同类问题再次出现

## 六、持续改进

每次项目迭代结束时：

1. **更新 AGENTS.md**：记录新发现的陷阱和规范
2. **更新检查脚本**：添加新的检查规则
3. **补充测试**：覆盖率目标 ≥ 40%
4. **回顾 AI 错误**：哪些问题 AI 反复犯？写进检查清单

```
项目经验 → 更新文档 → 下次 AI 更准 → 项目质量更高
         ↑                                    |
         └────────────────────────────────────┘
```
