## 1. 协议总览

CarLife 是百度推出的车机手机互联方案，采用 **6 通道 TCP Socket** 架构，使用 **Protobuf** 作为消息序列化格式。

### 1.1 核心特征

| 特征 | 说明 |
|------|------|
| 传输协议 | TCP Socket（6 个独立连接） |
| 序列化格式 | Protocol Buffers (proto3) |
| 字节序 | Big-Endian（网络字节序） |
| 视频编码 | H.264 (AVC)，可选 H.265 (HEVC) |
| 音频编码 | PCM / AAC / Opus |
| 默认分辨率 | 800×480 (WinCE) / 车机屏幕分辨率 |
| 帧率 | 30fps |
| 认证机制 | 可选，可绕过 |

### 1.2 角色定义

| 角色 | 全称 | 说明 | 典型设备 |
|------|------|------|---------|
| **HU** | Head Unit | 车机端（接收端） | WinCE 车机、本项目的 MdRole |
| **MD** | Mobile Device | 手机端（发送端） | 手机 B (CarWith)、本项目的 HuRole |

> ⚠️ **注意**：本项目中 MdRole 扮演 HU 角色（面向车机），HuRole 扮演 MD 角色（面向手机 B）。命名与标准协议角色相反，因为 MdRole/HuRole 是按"连接方向"命名的。

## 2. 通道架构

### 2.1 六通道定义

CarLife 协议定义了 6 个独立的 TCP Socket 通道：

```
┌──────────────────────────────────────────────────────────────┐
│                    CarLife 6 通道架构                         │
├──────────┬──────────┬────────────┬───────────────────────────┤
│ 通道     │ 方向     │ 端口(MD侧)│ 功能                      │
├──────────┼──────────┼────────────┼───────────────────────────┤
│ CMD      │ 双向     │ 7240       │ 控制信令（握手、认证、配置）│
│ VIDEO    │ MD → HU  │ 8240       │ 视频流（H.264 帧）         │
│ MEDIA    │ MD → HU  │ 9240       │ 媒体音频（音乐 PCM/AAC）   │
│ TTS      │ MD → HU  │ 9241       │ 导航语音（TTS 音频）       │
│ VR       │ HU → MD  │ 9242       │ 语音识别（麦克风数据）     │
│ CTRL     │ HU → MD  │ 9340       │ 触摸/按键控制事件          │
└──────────┴──────────┴────────────┴───────────────────────────┘
```

### 2.2 端口映射（ADB Forward 模式）

USB 有线连接时，车机端执行 `adb forward` 建立端口转发：

| 通道 | HU 本地端口 | MD 远程端口 | adb forward 命令 |
|------|------------|------------|-----------------|
| CMD  | 7200 | 7240 | `adb forward tcp:7200 tcp:7240` |
| VIDEO | 8200 | 8240 | `adb forward tcp:8200 tcp:8240` |
| MEDIA | 9200 | 9240 | `adb forward tcp:9200 tcp:9240` |
| TTS  | 9201 | 9241 | `adb forward tcp:9201 tcp:9241` |
| VR   | 9202 | 9242 | `adb forward tcp:9202 tcp:9242` |
| CTRL | 9300 | 9340 | `adb forward tcp:9300 tcp:9340` |

### 2.3 通道类型枚举

```kotlin
enum class ChannelType(val huPort: Int, val mdPort: Int) {
    HU_CMD(7240, 7200),     // CMD 通道
    HU_VIDEO(8240, 8200),   // 视频通道
    HU_MEDIA(9240, 9200),   // 媒体音频通道
    HU_TTS(9241, 9201),     // TTS 通道
    HU_VR(9242, 9202),      // VR 通道
    HU_CTRL(9340, 9300)     // 控制通道
}
```

### 2.4 通道类型编码

```cpp
CMD_CHANNEL   = 1   // 命令通道
VIDEO_CHANNEL = 2   // 视频通道
MEDIA_CHANNEL = 3   // 媒体音频通道
TTS_CHANNEL   = 4   // 导航语音通道
VR_CHANNEL    = 5   // 语音识别通道
CTRL_CHANNEL  = 6   // 触摸控制通道
```

## 3. 传输层协议格式

CarLife 协议有两种包头格式：**CMD 格式**（用于 CMD/CTRL 通道）和 **媒体格式**（用于 VIDEO/MEDIA/TTS/VR 通道）。

### 3.1 CMD 通道包头（8 字节）

用于 CMD 和 CTRL 通道的控制消息传输。

```
偏移    字段           大小     说明
────────────────────────────────────────────────
0-1     data_len      2B      Protobuf 数据长度 (Big-Endian)
2-3     reserved      2B      保留字段（值为 0）
4-7     service_type  4B      消息 ID / 服务类型 (Big-Endian)
8+     protobuf_data  变长     Protobuf 序列化数据
```

**字节布局**:
```
+--------+--------+--------+--------+--------+--------+--------+--------+
| data_len (高)   | data_len (低)   | reserved       | service_type    |
|  byte0 |  byte1 |  byte2 |  byte3 |  byte4 |  byte5 |  byte6 |  byte7 |
+--------+--------+--------+--------+--------+--------+--------+--------+
```

**解析代码**:
```kotlin
// 读取
val dataLen = ((header[0].toInt() and 0xFF) shl 8) or (header[1].toInt() and 0xFF)
val serviceType = ((header[4].toInt() and 0xFF) shl 24) or
                  ((header[5].toInt() and 0xFF) shl 16) or
                  ((header[6].toInt() and 0xFF) shl 8) or
                  (header[7].toInt() and 0xFF)

// 写入
header[0] = ((dataLen shr 8) and 0xFF).toByte()
header[1] = (dataLen and 0xFF).toByte()
header[4] = ((serviceType shr 24) and 0xFF).toByte()
header[5] = ((serviceType shr 16) and 0xFF).toByte()
header[6] = ((serviceType shr 8) and 0xFF).toByte()
header[7] = (serviceType and 0xFF).toByte()
```

### 3.2 媒体通道包头（12 字节）

用于 VIDEO、MEDIA、TTS、VR 通道的音视频数据传输。

```
偏移    字段           大小     说明
────────────────────────────────────────────────
0-3     data_len      4B      原始数据长度 (Big-Endian)
4-7     timestamp     4B      时间戳（毫秒）(Big-Endian)
8-11    service_type  4B      消息 ID / 服务类型 (Big-Endian)
12+     raw_data      变长     H.264 帧 / PCM 采样等
```

**字节布局**:
```
+--------+--------+--------+--------+--------+--------+--------+--------+
| data_len                                  | timestamp                   |
|  byte0 |  byte1 |  byte2 |  byte3 |  byte4 |  byte5 |  byte6 |  byte7 |
+--------+--------+--------+--------+--------+--------+--------+--------+
| service_type                              |
|  byte8 |  byte9 |  byte10|  byte11|
+--------+--------+--------+--------+
```

**解析代码**:
```kotlin
val dataLen = ((header[0].toInt() and 0xFF) shl 24) or
              ((header[1].toInt() and 0xFF) shl 16) or
              ((header[2].toInt() and 0xFF) shl 8) or
              (header[3].toInt() and 0xFF)
val timestamp = ((header[4].toInt() and 0xFF) shl 24) or
                ((header[5].toInt() and 0xFF) shl 16) or
                ((header[6].toInt() and 0xFF) shl 8) or
                (header[7].toInt() and 0xFF)
val serviceType = ((header[8].toInt() and 0xFF) shl 24) or
                  ((header[9].toInt() and 0xFF) shl 16) or
                  ((header[10].toInt() and 0xFF) shl 8) or
                  (header[11].toInt() and 0xFF)
```

### 3.3 USB 传输层包头（8 字节，AOA 单流复用模式）

当使用 USB AOA 连接时，6 个通道的数据复用同一条 USB 流，通过额外的 8 字节传输层包头区分通道：

```
偏移    字段           大小     说明
────────────────────────────────────────────────
0-2     reserved      3B      保留（值为 0）
3       msg_type      1B      通道类型 (1=CMD, 2=VIDEO, 3=MEDIA, 6=TOUCH)
4-7     payload_len   4B      消息体长度 (Big-Endian)
```

> ⚠️ 本项目使用 Wi-Fi TCP 连接（非 USB AOA），不使用此包头格式。

### 3.4 ChannelHeader 抽象（本项目实现）

本项目将两种包头统一为 `ChannelHeader` 密封类：

```kotlin
sealed class ChannelHeader {
    companion object {
        const val MAGIC: Short = 0x434C      // "CL" 魔数
        const val CMD_HEADER_SIZE = 8         // CMD 包头大小
        const val MEDIA_HEADER_SIZE = 11      // 媒体包头大小
    }

    // CMD 包头（8 字节）：magic(2) + payloadType(1) + payloadLength(4) + crc(1)
    data class Cmd(val payloadType: Int, val payloadLength: Int, val crc: Byte)

    // 媒体包头（11 字节）：magic(2) + payloadType(1) + timestamp(4) + payloadLength(4)
    data class Media(val payloadType: Int, val timestamp: Int, val payloadLength: Int)
}
```

> ⚠️ **注意**：`ChannelHeader` 是本项目内部的抽象格式（带 `magic` 字段），与标准 CarLife 协议的 CMD 8 字节 / 媒体 12 字节格式不同。实际网络传输使用 `readCarLifeMsg()` / `sendCarLifeMsg()` 方法，采用标准 CarLife 格式。

## 6. 消息 ID 完整映射表

### 6.1 消息 ID 编码规则

```
0x0001xxxx = CMD 通道消息
  0x000180xx = HU(车机) → MD(手机)
  0x000100xx = MD(手机) → HU(车机)
0x0002xxxx = VIDEO 通道消息
0x0003xxxx = MEDIA 通道消息
0x0004xxxx = TTS 通道消息
0x0005xxxx = VR 通道消息
  0x000580xx = HU → MD
0x0006xxxx = CTRL 通道消息
  0x000680xx = HU → MD
```

### 9.2 v2.3.0 协议版本说明（新增）

```kotlin
// VersionDetector 新增方法
fun isVersionAllowed(major: Int, minor: Int): Boolean {
    return major > 3 || (major == 3 && minor >= 2)
}

fun isVersionRecommended(major: Int, minor: Int): Boolean {
    return major >= 4
}

// ProtocolTranslator 版本转换
fun translateProtocolVersion(major: Int, minor: Int): Pair<Int, Int> {
    return when {
        major >= 4 -> Pair(4, 1)           // v4.1 推荐，直通
        major == 3 && minor >= 2 -> Pair(3, 2)  // v3.2 允许，直通
        major == 3 -> Pair(1, 0)           // v3.0/3.1 降级
        else -> Pair(1, 0)                 // 旧版直通
    }
}
```

### 10.2 ConnectionService 协调流程

```
ConnectionService
    │
    ├── startMdRole()
    │   ├── 启动 6 个 TcpServer (7200/8200/9200/9201/9202/9300)
    │   ├── mDNS 广播 (_carlife._tcp.)
    │   └── 等待车机连接
    │
    ├── startHuRole()
    │   ├── 创建 6 个 Channel 连接手机 B
    │   ├── 完成 CarLife 握手 (Phase 1-7)
    │   └── HuState → CONNECTED
    │
    ├── tryStartVideoAndAudioServices()
    │   ├── 检查 MdRole.READY && HuRole.CONNECTED
    │   ├── 请求 MediaProjection 授权
    │   ├── 启动 VideoService (MediaProjection + H.264 编码)
    │   ├── 启动 AudioService (AudioPlaybackCapture + AAC)
    │   └── 启动 DynamicBitrate (动态码率调节)
    │
    └── 数据转发
        ├── 手机 B VIDEO → MdRole.sendData(MD_VIDEO) → 车机
        ├── 手机 B MEDIA → MdRole.sendData(MD_MEDIA) → 车机
        ├── 车机 CTRL → HuRole.sendControlMsg() → 手机 B
        └── 车机 VR → HuRole.sendVrData() → 手机 B
```

## 附录 D: CarProjection 对比分析

| 维度 | CarProjection (参考) | CarLifeWirelessBox (本项目) |
|------|---------------------|---------------------------|
| **连接方式** | USB Accessory (有线) | WiFi TCP (无线) |
| **设备角色** | 仅 MD (手机侧) | 双角色: HuRole (手机B侧) + MdRole (车机侧) |
| **视频采集** | MediaProjection + H.264 | 同上 |
| **音频采集** | AudioPlaybackCapture | 同上 + AAC 编码 |
| **触摸注入** | AccessibilityService | 同上 |
| **协议握手** | 完整 CarLife 协议 (MD侧) | HuRole 完整 + MdRole 完整 |
| **USB 通道** | UsbManager.openAccessory() | USB Tethering TCP (非 AOA) |

### CarProjection 实现要点

1. **USB Accessory 通道**: `UsbManager.openAccessory()` → `FileInputStream`/`FileOutputStream`，6 通道复用单条 USB 流
2. **USB 传输层包头 (8 字节)**: `[reserved(3B)][msg_type(1B)][payload_len(4B)]`
3. **触摸处理**: 竖屏时计算画面偏移量 `leftX = (carWidth - portraitCarWidth) / 2`，使用不同缩放因子
4. **音频**: AudioPlaybackCapture (Android 10+) 或 REMOTE_SUBMIX (旧版本)
